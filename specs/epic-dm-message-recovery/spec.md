# DM Message Recovery & Multi-Format Inbound

Feature request specification for a defect-driven epic that closes the
"bell rings but the message never appears" failure mode in 1:1 direct
messages, hardens the bell against the post-NIP-17 transport split, and
adds a one-shot recovery + upgrade pass for messages already on the
relay or already in IndexedDB.

This document is the input to `/base:feature` and is structured the same
way as `specs/marmot-application-rumor-dispatch.md` and
`specs/emoji-feature-portable-spec.md`: spec body, decisions, acceptance
criteria. Per-story scaffolding will be generated by the planner once
the spec is approved.

---

## 1. Problem statement

### 1.1 Reported symptom

A user receives a 1:1 DM. The notification bell badge increments,
indicating that *some* event addressed to the local pubkey arrived on
the relay set. Opening the corresponding contact chat does **not**
display the message. The peer confirms they sent it. The defect is
non-deterministic from the user's point of view because it depends on
which transport (and which client) the peer used.

### 1.2 Three independent gaps

The DM pipeline currently has three latent gaps that can each drop a
message silently. They are listed in the order the user is most likely
to encounter them.

| ID | Gap | Where | Effect |
|----|-----|-------|--------|
| G1 | Strict envelope parser rejects bare-plaintext NIP-04 | `app/src/lib/directMessages.ts:98–123` (`parseDirectPayload`) | DM from a non-quizzl client over kind-4 silently dropped after decrypt; bell rings (no decrypt needed) |
| G2 | Bell watcher does not subscribe to NIP-17 gift wraps | `app/src/lib/directMessageNotifications.ts:36–39` (`subscribeDirectMessageNotifications`) | quizzl→quizzl DM does not bump the bell; if the chat is closed and the relay does not redeliver post-EOSE, the message is unobservable |
| G3 | No historical fetch for kind-1059 on chat mount | `app/src/components/contacts/ContactChat.tsx:128–146` (`init`) | A gift-wrapped DM that arrived while the chat was closed may not replay when the chat is opened, depending on relay redelivery semantics |

There is also a smaller, purely-local concern: messages whose stored
`content` is the raw JSON envelope string (`{"type":"text","text":"…"}`)
rather than the decoded text would render incorrectly. This is rare in
practice but cheap to defuse; see §3.4.

### 1.3 Why the recent refactors did not fix it

Commit `b32155f` ("unify MLS rumor dispatch via single subscriber +
handler registry") consolidated the **MLS group** rumor dispatch path.
It did not touch `directMessages.ts`, `ContactChat.tsx`, or
`directMessageNotifications.ts`. Commit `4df6284` is a Makefile change.
The DM pipeline's two-track structure (kind-4 inbound forever per D9a;
kind-1059 inbound+outbound per D9b, both established in `3dcc481`) is
unchanged.

### 1.4 Scope of supported DM message types

The fix must support every shape the live wire and the local store can
present:

- **kind-4 / NIP-04, JSON envelope** (`{"type":"text","text":…}` /
  `{"type":"image","version":1,…}`) — quizzl-to-quizzl legacy outbound,
  accepted today.
- **kind-4 / NIP-04, bare plaintext** — every other Nostr client
  (Damus, Amethyst, Coracle, 0xchat, Iris, Snort, Nostrudel, Primal,
  Plebeian Market, …). Currently dropped.
- **kind-1059 → seal 13 → rumor kind 14, JSON envelope** — quizzl-to-
  quizzl since `3dcc481`. Accepted today **only** while the chat is
  mounted and the relay redelivers.
- **kind-1059 → seal 13 → rumor kind 14, bare plaintext** — third-party
  NIP-17 clients that do not adopt quizzl's JSON envelope. Currently
  dropped on display.
- **kind-1059 → seal 13 → rumor kind 7** — DM reactions (story-07 of
  the emoji epic). Out of scope for this epic; already covered.
- **Legacy stored ChatMessage rows** — anything ever written into
  `quizzl:messages:dm:<peer>` regardless of original transport. Must
  render. If the row is malformed, replace via re-fetch.

## 2. Goals

1. **Display every DM** the user can decrypt. The display path must not
   reject a successfully-decrypted message just because its content
   shape differs from quizzl's JSON envelope.
2. **Ring the bell for every inbound DM** the local key can receive,
   regardless of transport (kind-4 or kind-1059).
3. **Recover historical DMs** that landed during a window in which the
   user was offline / the chat was closed / a gap above silently
   dropped them. After the fix lands, opening the chat should
   re-fetch from the relay set and surface anything previously missed.
4. **Self-heal local storage** for any persisted ChatMessage that was
   written in a malformed/legacy shape, by either upgrading the row
   in place or re-fetching the canonical event and replacing the row.

### Non-goals

- Adding outbound NIP-04 support. The NIP-17 outbound migration in
  `3dcc481` is intentional and stays.
- Cross-protocol attribution heuristics on the bell. The bell only
  needs to count; per-peer attribution comes from the open-chat path.
- Changing how reactions (kind-7 over gift wrap) are processed.
- Generalising to a unified inbound dispatcher across DMs and groups.
  The marmot-side dispatcher already exists; pulling DMs into it is a
  separate refactor.

## 3. Specification

### 3.1 Lenient inbound payload parser (closes G1)

`parseDirectPayload(raw: string)` becomes total: it never returns
`null` for a successfully-decrypted, non-empty string. Parsing
priority:

1. `JSON.parse(raw)` succeeds **and** matches the text-envelope shape
   → return `{ content: parsed.text }`.
2. `JSON.parse(raw)` succeeds **and** matches the image-envelope shape
   → return `{ content: buildImageMessageContent(parsed.caption),
   attachments: parsed.attachments }`.
3. `JSON.parse(raw)` succeeds but matches no known shape → return
   `{ content: raw }` (treat the JSON string itself as text).
4. `JSON.parse(raw)` throws → return `{ content: raw }`.

`raw === ''` (empty after decrypt) returns `null` so that obviously
broken events are still skipped.

`decryptDirectPayload` is unchanged in signature; only its parser leaf
becomes lenient. `ingestEvent` in `ContactChat.tsx` continues to skip
on `null` only, which now means *only* "decrypt failed or content was
empty" — never "envelope shape mismatch".

The same parser is also used in the kind-1059 path
(`ContactChat.handleGiftWrapEvent` line 209), so this single change
fixes inbound for both legacy NIP-04 plaintext **and** any future
NIP-17 client that sends bare plaintext inside the kind-14 rumor.

### 3.2 Bell watcher subscribes to gift wraps (closes G2)

`subscribeDirectMessageNotifications` adds a second subscription:

```ts
const giftWrapSub = ndk.subscribe({ kinds: [1059], '#p': [ownPubkeyHex] });
```

The handler unwraps via `unwrapAndOpen(evt, selfPrivKeyHex)`. On
success, if `rumor.kind === 14` **and** `rumor.pubkey !== ownPubkey`,
proceed exactly like the kind-4 path: dedup on `rumor.id`, check
last-read, `rememberContact(rumor.pubkey)`, `incrementDirectMessage`.

Failures (decryption errors, non-DM rumors such as kind-444 / kind-21059
welcome and join-request rumors that share the kind-1059 carrier) are
silently swallowed — the same pattern `ContactChat.handleGiftWrapEvent`
already uses. The kind-1059 carrier is shared with marmot welcomes and
join requests, so silent-discard on unwrap failure is required, not a
shortcut.

The `seen` set already de-dupes by event id; it must move to keying by
**inner rumor id** for the gift-wrap path (the outer wrap id changes
per delivery; the rumor id is stable). Add a parallel
`seenRumorIds: Set<string>` so the kind-4 path is unaffected.

`DirectMessageNotificationsWatcher` requires `privateKeyHex` to do the
unwrap. It already has access to it via `useNostrIdentity()` and
already passes it into `connectNdk`. No additional plumbing needed.

### 3.3 Historical fetch + replay for kind-1059 (closes G3)

In `ContactChat.init`, immediately after the kind-4 historical fetch,
add a parallel fetch:

```ts
const giftWrapHistorical = await fetchEventsWithTimeout(ndk, {
  kinds: [1059],
  '#p': [pubkeyHex],
  limit: 500,
});
```

(`limit: 500` because gift wraps from any sender share this filter —
welcomes, join requests, DMs from peers other than the current one.
The fan-out is bounded by relay-set size and recent activity.)

Each event runs through the same handler shape used for live gift
wraps:

- `unwrapAndOpen` → silent skip on failure
- `shouldIngestRumor(rumor, peerPubkeyHex)` → silent skip otherwise
- `rumor.kind === 14` only (kind-7 is processed by the existing
  reactions path)
- `parseDirectPayload(rumor.content)` (now lenient per §3.1)
- `appendMessage(threadId, msg)` — `appendMessage` already de-dupes
  by id, so a wrap that was both historically fetched and live-
  delivered yields one row.

Limit `500` is a starting value. If profiling shows relay-side fan-out
is large for active accounts, tighten it; the bell-watcher's global
gift-wrap subscription will still pick up live arrivals.

### 3.4 Local store self-heal pass

`loadMessages(threadId)` for DM threads gets a one-shot post-process
that detects malformed rows and either **upgrades** them in place or
marks them for **refetch-and-replace**.

A row is malformed if **any** of the following is true:

1. `content` matches `^\s*\{\s*"type"\s*:\s*"(text|image)"` —
   indicates the row was written with the JSON envelope still embedded
   in `content` instead of decoded.
2. `id` is not a 64-character lowercase hex string — indicates it is
   neither a NIP-01 rumor id nor a kind-4 event id, and was probably a
   client-side temp UUID that was never reconciled.
3. For an `attachments`-bearing row, `attachments.full?.url` is missing
   while `attachments.full?.sha256` is present — indicates an
   optimistic image row whose upload swap never completed.

Recovery strategy:

- **Case 1 (envelope-in-content)**: re-run the leaf of
  `parseDirectPayload` over the stored `content` and rewrite the row
  in place. No network round trip required. Idempotent.
- **Case 2 (non-canonical id)**: enqueue for refetch; if a re-fetch
  yields a canonical-id row, upsert it and delete the malformed one.
  If no canonical replacement is found within the fetch window,
  leave the malformed row in place (still rendered; user does not
  lose data). Logged at info level.
- **Case 3 (orphaned optimistic image)**: if the message originator
  was the local user, drop the row (it never landed); otherwise treat
  as Case 2.

The self-heal pass runs once per thread, on the first
`loadMessages(threadId)` after the feature ships. Persist a per-thread
"healed" marker (`localStorage` key `lp_dmHealed_v1` keyed by thread)
so subsequent loads skip the scan.

### 3.5 Order of operations on chat mount

After this epic, `ContactChat.init` must complete in this order so the
user sees a stable UI:

1. `loadMessages(threadId)` from IDB; run self-heal pass (§3.4) before
   `setMessages(stored)`.
2. `setMessages(stored)` so something renders immediately.
3. Fire kind-4 historical fetch (existing).
4. Fire kind-1059 historical fetch (§3.3) **in parallel** with #3.
5. Ingest both result sets through the lenient parser (§3.1) and
   `upsertMessages`.
6. Subscribe live to kind-4 (existing) and kind-1059 (existing).

Steps 3 and 4 are independent; step 5 must wait for both to settle so
the message order in the rendered list is monotonic.

### 3.6 Decryption-failure observability

Add structured info-level logs (using the namespaced logger introduced
in `77c5d44`) at every silent-skip site:

- `parseDirectPayload` returning the literal `{ content: raw }`
  fallback (counts as "kept, lenient").
- `decryptDirectPayload` throwing (counts as "decrypt failed").
- `unwrapAndOpen` throwing (counts as "not for me / not a DM").
- `shouldIngestRumor` returning false (counts as "wrong peer").

These are info-level, not error-level — the silent-skip is intentional
in many cases (gift wraps for other conversations). The point is to
make future regressions debuggable without re-discovering the silent
drop the way this defect was discovered.

## 5. Decisions

### D1 — Lenient parser fallback policy

The fallback when `parseDirectPayload` cannot match a known envelope
is `{ content: raw, attachments: undefined }` — i.e., *the raw
decrypted string is shown to the user verbatim*. This means a
third-party NIP-04 image post that uses a different attachment scheme
(e.g., URL-in-body) renders as text containing that URL, which the
existing linkifier (`dcc0f33`) will turn into a clickable link. Good
enough; we explicitly do not try to interpret foreign attachment
schemes.

**Why:** the failure mode "user sees no message" is strictly worse
than "user sees a URL they can click". The fallback is also the same
shape every other Nostr client renders.

### D2 — No outbound NIP-04 fallback

We do **not** re-introduce NIP-04 outbound to talk to legacy peers.
Inbound is forever; outbound is NIP-17 only. Peers on legacy clients
who reply to our NIP-17 messages will be invisible to us in the
unrealistic case that they cannot decrypt kind-1059. In practice the
ecosystem has converged on dual-listen.

### D3 — Bell attribution for gift wraps

The bell *does* unwrap gift wraps (in the watcher), so attribution to
a specific peer is preserved. We considered a cheaper "ring without
attribution" path (filter on kind-1059 to self, increment a generic
counter) but rejected it: the dropdown lists per-contact unread
counts, so attribution must be available at increment time.

The unwrap is cheap (NIP-44 is a single ECDH + AES-GCM-style decrypt;
NIP-17 layers two of them). We are already doing this in
`ContactChat.handleGiftWrapEvent` for every live wrap.

### D4 — Historical fetch limits

Kind-4 historical fetch limit stays at 200 (existing). Kind-1059
historical fetch limit is 500. The asymmetry reflects that kind-4
filtered by `authors:[peer]` returns DMs from one peer only; kind-1059
filtered by `'#p':[self]` returns gift wraps from *any* sender —
welcomes, join requests, DMs from other peers. We must over-fetch and
discard.

### D5 — Self-heal scope

Self-heal runs **only on DM threads** (`groupId.startsWith('dm:')`).
Group-message persistence has its own format and is not touched by
this epic.

The healed-marker is per-thread, not global, so adding a new contact
later does not skip the heal pass for the new thread.

### D6 — Logging policy

Silent-skip sites log at **info** level, never warn or error. Decrypt
failures on gift wraps are normal (other-conversation traffic). A
bumped error rate in dashboards must continue to mean "something is
actually wrong"; this epic must not produce false-positive noise.

### D7 — Dedup keys

| Path | Dedup key | Why |
|------|-----------|-----|
| ContactChat IDB store | `ChatMessage.id` (rumor id for kind-1059, event id for kind-4) | Already in place; cross-protocol dedup is impossible (different ids per transport) and is therefore not attempted |
| Bell watcher kind-4 | event id | Outer wrap is the event itself for legacy DMs |
| Bell watcher kind-1059 | inner rumor id | Outer wrap pubkey is ephemeral per NIP-59; outer id changes per delivery |

A single `Set<string>` per watcher instance is fine; ids are
collision-resistant across the two namespaces because rumor ids are
hashes of canonical rumors and event ids are hashes of canonical
events with disjoint content shapes.

### D8 — No backfill from external archives

If a message landed on a relay that the user has since removed from
their relay set, this epic does not attempt to recover it. The user
configured the relay set; we honour it.

## 6. Acceptance criteria

ACs are stable identifiers that survive across refactors. They are
the contract `/base:feature` measures each story against.

### Lenient parser (§3.1)

**AC-01** `parseDirectPayload('hello')` returns `{ content: 'hello' }`.

**AC-02** `parseDirectPayload('{"type":"text","text":"hi"}')` returns
`{ content: 'hi' }`.

**AC-03** `parseDirectPayload('{"unknown":"shape"}')` returns
`{ content: '{"unknown":"shape"}' }`.

**AC-04** `parseDirectPayload('')` returns `null`.

**AC-05** `parseDirectPayload` for a valid image envelope returns the
same value as today (no behavior change on the happy path) — covered
by the existing `directMessages.test.ts` image round-trip.

**AC-06** A unit test in `app/tests/unit/directMessages.test.ts`
asserts AC-01 through AC-04 by direct call.

### ContactChat inbound (§3.1, §3.3, §3.5)

**AC-07** `ingestEvent` in `app/src/components/contacts/ContactChat.tsx`
returns a `ChatMessage` (not `null`) when the kind-4 event decrypts
to a non-empty bare-plaintext string.

**AC-08** `handleGiftWrapEvent` in `ContactChat.tsx` upserts a
`ChatMessage` for any kind-14 rumor whose decrypted content is
non-empty, regardless of envelope shape.

**AC-09** `ContactChat.init` performs a `fetchEventsWithTimeout({
kinds: [1059], '#p': [pubkeyHex], limit: 500 })` call in addition to
the existing kind-4 fetches, and ingests every resulting event
through the same gift-wrap unwrap+parse path used for live wraps.

**AC-10** `ContactChat.init` waits for both historical fetches to
settle before calling `upsertMessages`, so the rendered message list
is in `createdAt` order regardless of arrival order.

**AC-11** A historical fetch that yields a gift wrap whose inner
rumor id matches a `ChatMessage` already in IDB does not produce a
duplicate row (`appendMessage`'s id-based dedup is exercised by the
historical path).

**AC-12** An e2e test `app/tests/e2e/dm-third-party-inbound.spec.ts`
publishes a kind-4 NIP-04 event with bare-plaintext encrypted content
to the local user, asserts the bell badge becomes 1, opens the chat,
and asserts the bubble renders the plaintext.

**AC-13** An e2e test `app/tests/e2e/dm-historical-recovery.spec.ts`
seeds 5 historical gift-wrapped DMs on the relay before the
recipient's session starts, opens the contact chat, and asserts all 5
render in `created_at` order with no duplicates.

### Bell watcher (§3.2)

**AC-14** `subscribeDirectMessageNotifications` accepts a
`privateKeyHex: string` parameter in addition to `ndk` and
`ownPubkeyHex`.

**AC-15** `subscribeDirectMessageNotifications` opens two
subscriptions: `{ kinds: [4], '#p': [ownPubkeyHex] }` and
`{ kinds: [1059], '#p': [ownPubkeyHex] }`.

**AC-16** A kind-1059 event whose unwrapped rumor has `kind === 14`,
`pubkey !== ownPubkeyHex`, and `created_at * 1000 > getDirectMessage
LastReadAt(rumor.pubkey)` calls `rememberContact(rumor.pubkey)` and
`incrementDirectMessage(rumor.pubkey)` exactly once per rumor id.

**AC-17** A kind-1059 event whose unwrap throws does not call
`rememberContact` or `incrementDirectMessage`. No unhandled promise
rejection escapes the handler.

**AC-18** A kind-1059 event whose unwrapped rumor has `kind !== 14`
(e.g., kind-7 reaction, kind-444 welcome, kind-21059 join request)
does not call `rememberContact` or `incrementDirectMessage`.

**AC-19** A kind-1059 event re-delivered with a different outer id
but the same inner rumor id increments the bell exactly once.

**AC-20** `DirectMessageNotificationsWatcher` passes
`privateKeyHex` (already available via `useNostrIdentity()`) to
`subscribeDirectMessageNotifications`.

**AC-21** A unit test in
`app/tests/unit/directMessageNotifications.test.ts` covers AC-15
through AC-19 against a fake NDK that emits hand-crafted gift-wrap
events.

**AC-22** An e2e test `app/tests/e2e/dm-giftwrap-bell.spec.ts`
asserts that a NIP-17 DM sent from a second browser context
increments the bell badge in the recipient's context **without** the
recipient opening the chat.

### Local-store self-heal (§3.4)

**AC-23** `loadMessages(threadId)` for `threadId` starting with
`'dm:'` runs the self-heal pass exactly once per thread per device.
A `localStorage` entry under key `lp_dmHealed_v1` records the set of
healed thread ids.

**AC-24** A row whose `content` matches `/^\s*\{\s*"type"\s*:\s*"(
text|image)"/` is rewritten in place: `content` becomes the decoded
text, and (for image envelopes) `attachments` is populated from the
embedded JSON. The row's `id` and `createdAt` are unchanged.

**AC-25** A row whose `id` is not a 64-character lowercase hex string
is enqueued for refetch. If a kind-4 or kind-1059 historical fetch
yields a canonical-id replacement, the malformed row is deleted via
`appendMessage` of the replacement and a follow-up filter (the
existing `byId.set` upsert in `upsertMessages` is sufficient because
the replacement has a different id; the malformed row must be dropped
explicitly by id list before the upsert).

**AC-26** A row authored by `pubkeyHex` (own messages) with an
`attachments.full.sha256` but no `attachments.full.url` is removed.

**AC-27** The self-heal pass is idempotent: a second `loadMessages`
call returns the same result and does not re-rewrite rows.

**AC-28** A unit test in
`app/tests/unit/directMessages/selfHeal.test.ts` asserts AC-24,
AC-26, and AC-27.

**AC-29** An e2e test `app/tests/e2e/dm-self-heal.spec.ts` seeds an
IDB row whose `content` is the raw JSON envelope string, opens the
chat, asserts the bubble renders the decoded text, reloads the page,
and asserts the bubble still renders the decoded text. After reload
the test inspects IDB via the test bridge and asserts the row has
`content === '<decoded>'`.

### Observability (§3.6)

**AC-30** Each silent-skip site
(`parseDirectPayload` lenient fallback, `decryptDirectPayload`
throwing, `unwrapAndOpen` throwing, `shouldIngestRumor` returning
false) calls the namespaced logger from `app/src/lib/logger.ts` (or
the project's equivalent introduced in `77c5d44`) at `info` level
with a stable message tag.

**AC-31** No silent-skip site calls `console.warn` or
`console.error`. The unit tests assert that the test logger receives
`info` calls only.

### Cross-cutting

**AC-32** No outbound NIP-04 events are published by any code path
introduced in this epic. The existing `publishDirectMessage` (NIP-17
gift-wrap only) is not modified.

**AC-33** All user-visible strings introduced by this epic (none
expected on the happy path; possibly an info-level toast for the
self-heal "we recovered N missing messages" case if §3.4 produces a
non-empty refetch set) are added to both `en` and `de` blocks of
`app/src/lib/i18n.ts`, per the project translations rule in
`CLAUDE.md`.

## 7. Story breakdown (proposed)

The planner is free to repartition; this is the natural cleavage.

1. **Story 01 — Lenient parser.** §3.1, AC-01 through AC-08, AC-12,
   AC-30, AC-31. Smallest, highest-impact, lands first.
2. **Story 02 — Bell gift-wrap subscription.** §3.2, AC-14 through
   AC-22, AC-30, AC-31.
3. **Story 03 — Historical kind-1059 fetch.** §3.3, §3.5, AC-09
   through AC-13.
4. **Story 04 — Local-store self-heal.** §3.4, AC-23 through AC-29.
5. **Story 05 — E2E hardening.** Pure test additions for the gaps
   `notification-bell.spec.ts` does not cover; folds into earlier
   stories if the planner prefers.

Stories 01–03 unblock the user-facing failure mode. Story 04 is a
pure local cleanup and can ship on a slower cadence.

## 8. Risks

- **Relay redelivery semantics.** Some relays do not redeliver
  kind-1059 to a re-subscribing client. The historical fetch in §3.3
  is the primary mitigation; if it returns nothing, messages from the
  pre-fix window are unrecoverable. The bell-watcher fix in §3.2 only
  helps from the moment it lands forward.
- **Cross-protocol non-dedup.** A single logical message *cannot* be
  deduplicated when received over both NIP-04 and NIP-17 because the
  ids are computed from disjoint canonical inputs. If a peer's client
  sends both — unusual, but not impossible during their own
  migration — the recipient will see the message twice. Acceptable
  trade-off; documented in §4.1.
- **Self-heal misclassification.** A genuine plaintext message whose
  text *happens* to start with `{"type":"text"` would be misclassified
  by AC-24's regex and "decoded" incorrectly. Probability is
  vanishingly small but not zero. Mitigation: the regex requires the
  full JSON shape including `"text"` or `"image"` after `"type":`;
  bare `{"type":"foo"` does not match.
- **Bell unwrap CPU cost.** Every kind-1059 addressed to the local
  pubkey now triggers a NIP-44 decrypt in the watcher, even if the
  user never opens the originating chat. Welcomes and join requests
  share this carrier, so the cost was already being paid by
  `ContactChat` when open and by `welcomeSubscription` always; the
  marginal cost of the bell-watcher path is ~one decrypt per inbound
  wrap. Negligible at observed message rates.
